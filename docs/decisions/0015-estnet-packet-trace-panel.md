# ADR 0015: ESTNeT packet-trace panel — opt-in 2D time-series track

## Status

Accepted (2026-07-02). Live on branch `estnet-trace-pipeline` only; **not
merged to `main`** (see decision 8).

## Context

After the multi-beam SLS track was retired (ADR 0013), the validated
replacement direction was **selected-path realism**: a packet-level
time-series (latency / jitter / throughput / loss / handover) for the demo's
GS-A → satellite → GS-B path, generated by the customer-named external
simulator (ESTNeT/OMNeT++, published Würzburg v1.0, used unmodified) and by
the requirement-named network test tools (ping / iperf3). The requirement
side has not delivered its own trace data yet; this track builds the entire
delivery-ready pipeline and rehearses it with our own generated traces,
honestly labeled.

Two prior lessons bound the design:

- **Scale/canvas fit** (ADR 0013): per-path packet metrics are a time-series,
  not global geometry — they belong in a 2D panel, never painted onto the
  globe. The one attempt to animate "physically real flow" on the globe's
  stylized orbit-render layer failed and was reverted; that layer is a
  viewer-owned display projection, not source truth.
- **Opt-in overlays never replace the accepted surface** (the standing
  delivery-safety rule R11, re-affirmed when ADR 0013 retired the SLS
  overlay): the customer-accepted single-link surface (19/19) must stay
  byte-identical by default.

## Decisions

1. **Opt-in only, add-only.** The packet-trace section renders in the V4 side
   panel only when the ESTNeT display mode is on (toolbar toggle, persisted in
   `localStorage`; a `?estnet=1` deep link seeds the mode in-memory for that
   load only). Default = OFF renders nothing and leaves the accepted 19/19
   surface untouched. Turning it on only ADDS one disclosure section; turning
   it off removes the section's DOM entirely. Gate:
   `npm run verify:estnet:panel` (fresh-profile CDP browser gate — fresh
   profile is load-bearing because the mode persists; also asserts add-only
   via structural signature, menu selectability, axis semantics via
   `data-y-axis`, verbatim `assumptionSet`/`nonClaims`, pair binding, and
   zero console errors). `verify:g1` (19/19) is re-run after every
   panel-touching change.

2. **2D panel, never the globe.** All packet-trace data renders as an SVG
   time-series inside the panel section (plus two standalone preview pages).
   Nothing from this track draws on the globe.

3. **Provenance ladder, fail-closed.** Every trace fixture carries
   `sourceClass` (`external-simulator-derived` | `requirement-provided-estnet`
   | `network-test-derived`), `assumptionSet`, and mandatory `nonClaims`
   including "not operator-measured". The ingest tool REFUSES the
   `operator-measured` tier and refuses format×provenance mislabels (a ping
   capture cannot claim to be an ESTNeT artifact). Manifest entries can never
   upgrade a fixture's tier; the panel renders provenance from the fixture
   itself. Simulation agreement with the viewer model is labeled
   implementation consistency, never validation against a measurement (R12).

4. **Reproducible generator → sim → adapter → fixture chain.** Scenarios are
   tracked in `scripts/estnet/scenario/` (the multi-phase domestic scenario is
   GENERATED by `generate_handover_scenario.mjs` from the viewer's own
   projection — at 25 phases, hand-wiring node/app cross-maps is the
   silent-loss failure mode). `npm run estnet:regen` re-runs everything and
   diffs the committed fixtures (regen must be byte-stable);
   `npm run verify:estnet` gates the contract + manifest + ingest parsers;
   `npm run estnet:crosscheck` re-derives every delivered sample's latency
   from independent python-sgp4 geometry (dual-model consistency) and checks
   MEO/LEO segments against independently computed co-visibility.
   `estnet:regen` re-verifies the generated scenario's phase table with
   independent python-sgp4 (`verify_handover_phases_independent.py`) before
   running its sim; the hand-authored CHT×SANSA phase table was locked by
   the precheck pair (`precheck_2hop_geometry.mjs` + `_independent.py`).

5. **Mask-trim finding (honest boundary).** The viewer's visibility grid is
   30 s; a LEO can drop below a station mask INSIDE the final grid step while
   the policy still serves it to the next grid instant. Serving phases keep
   the viewer's geometry-true boundaries, but each phase's SEND window is
   trimmed (1 s sampling, 2 s guard) so every transmitted packet is
   continuously above both masks — disclosed in the fixture `nonClaims`, and
   the crosscheck enforces it.

6. **Route↔trace pair binding.** Manifest entries carry pair HINTS
   (`stationA`/`stationB` registry ids) mirroring the fixture's declared
   endpoints; `verify:estnet` enforces hint == fixture metadata == registry
   id. The panel pre-selects the route's own trace (tie-breaker:
   `default: true`, then manifest order); a cross-pair trace hides the
   viewer-model overlay (fail-closed, unknown routes included) and renders an
   explicit one-line disclosure — never a silent omission.

7. **Requirement-data readiness.** `PACKET-TRACE-CONTRACT.md` +
   `external_trace_ingest.py` (`.vec` / scavetool CSV / ping / iperf3) keep
   the ingest → stage → review → curated-fixture-commit path armed for the
   pending requirement-side delivery; the committed loopback ping/iperf3
   fixtures are the exercised rehearsal of that path (machine-local tier,
   disclosed).

8. **Merging to `main` requires an independent, explicit decision.** The
   standing rule is: work stays on its branch, pushed to origin; no merge,
   no fast-forward, and no re-asking — `main` is the delivery-safe accepted
   surface and the user gates what lands there on their own schedule. Nothing
   in this track pre-authorizes that merge.

## Consequences

- The accepted 19/19 surface stays untouched by default — proven by gates,
  not by intent (`verify:estnet:panel`, `verify:g1`).
- A future requirement-side trace delivery reduces to: ingest → review →
  commit fixture + one manifest line (with pair hints); the panel, gates,
  honesty rails, and reconciliation block pick it up without code changes —
  subject to the parser gate (an unlisted format still means adapter work).
- Every committed trace remains regenerable and independently cross-checked;
  provenance can be audited from the fixture alone.
