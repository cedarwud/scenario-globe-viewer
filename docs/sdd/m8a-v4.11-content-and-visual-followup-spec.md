# M8A-V4.11 Content + Visual Followup Spec

Date: 2026-05-04
Amended: 2026-05-06
Status: **v2 amended after review (review doc:
m8a-v4.11-content-visual-spec-review.md)**. Doc-only spec, not
implementation prompt, not new phase authority. Stays inside V4.11
followup framework (no V4.12 opening). Supersedes the
panel-surface-only direction of the unwritten r3 prompt.

## 0. Why this spec exists

Visual pass r2 (commit 764629b) raised actual visual quality only
modestly because it applied design tokens but did not change panel
composition or content distribution. r2 self-estimate (7.5/10) was
control-side overoptimism; honest evaluation against vc3e reference
puts r2 at ~5/10 average, with only Evidence tab transformed.

Two follow-on conversations independently identified the same root:
**the inspector content architecture is wrong**, not just the styling.
This spec merges both diagnoses and resolves into a single content +
visual followup direction.

This is doc-only. No implementation prompt is generated by accepting
this spec. A separate implementation prompt will follow only after
spec acceptance.

## 1. Diagnosis, amended after review

### 1.1 Density and layering are both problems

V4.11 mixes six concern types into similar visual weight:
- customer requirement readouts
- Truth boundary disclosures
- Source provenance / evidence
- Operator decision context
- Real-time scene state
- Internal V4.10/V4.11 contract defense language

The v1 sentence "Information density is not the problem; layering is"
was too absolute. Density is part of the problem because unclear
ownership makes too much content visible by default. The v2 rule is:
**density and layering are both problems; density is caused by unclear
ownership and default visibility.**

Default-content reduction criteria:
- first read answers "what is happening / why / next" without panel
  scrolling at 1440x900;
- long tables move behind Archive or explicit disclosure;
- every boundary subtype has one owner surface;
- narrow 390x844 and tablet 768x1024 do not stack top-strip, rail,
  inspector, footer, and chips into competing text columns;
- vc3d/vc3e comparison scores figure-ground separation, first-read
  time, panel density, narrow behavior, and benchmark delta.

### 1.2 Content distribution must differ by window

V4.11 inspector currently applies the same 4-tab schema to W1-W5 and
the same prose row layout to the left rail. Each V4.6D window has
different operator concerns; the same schema cannot answer all of
them. The Slice 0 storyboard rewrite proposal already named per-window
operator questions. V2 makes those questions the content contract.

### 1.3 Replay clock continues during inspection

When a reviewer opens Details, replay continues to advance through
V4.6D windows. The content being read can become stale before reading
finishes. Current controller facts verified for v2:
- `M8aV4GroundStationSceneController` exposes `pause()`.
- Details open currently sets `detailsDisclosureOpen = true` and
  `activeInspectorTab = "decision"`; it does not pin replay state.
- Existing final-hold logic pauses at the end of the sequence.

Reviewer mode is therefore a real runtime state-machine requirement,
not a copy or CSS tweak. §7 defines the implementation contract.

### 1.4 Color semantics must preserve orbit identity tokens

V1 proposed a new cyan/blue/amber orbit palette. That conflicts with
current centralized V4.11 tokens. V2 rescinds that proposal. Existing
orbit token meanings must be preserved:
- `--m8a-v411-orbit-leo: var(--m8a-v411-ui-amber)` (`#FBBF24`)
- `--m8a-v411-orbit-meo: var(--m8a-v411-ui-mint)` (`#5EEAD4`)
- `--m8a-v411-orbit-geo: var(--m8a-v411-ui-cyan)` (`#38BDF8`)

State semantics move to a separate state token family (§6).

### 1.5 Narrow viewport is a designed surface, not fallback

Current narrow CSS hides the top strip and handover rail at `<=720px`,
keeps the inspector as a bottom sheet with `max-height: 28vh`, and
truncates footer chips. This is graceful degradation, not a delivered
review surface. V2 treats narrow and portrait tablet as separate UI
work with modal/drawer/focus contracts (§8).

### 1.6 Boundary content has separate subtypes

Boundary tab's entries are not co-equal:
- Scale boundary: valuable customer scope context.
- Endpoint boundary: valuable precision context.
- Simulation boundary: ambient disclosure.
- Service-layer boundary: ambient disclosure.
- Validation boundary: delivery/readiness status, not operator content.

V2 removes the standalone Boundary tab and assigns one owner per
boundary subtype (§4.4).

## 3. Product family and per-window framing

V4.11 is an **industrial mission-ops / satellite handover review
console**. It is not a marketing hero, debug HUD, generic glass
dashboard, or live operator controller. The visual system should be
quiet, dense enough for review, and explicit about scope. It should
feel like a satellite handover review surface that has been edited for
comprehension.

Five operator questions drive content. These are review-mode questions,
not live-control instructions.

| Window | Role label | Main question | Supporting question |
|---|---|---|---|
| W1 `leo-acquisition-context` | focus | LEO 品質夠不夠、能用多久？ | 服務時長 + 品質類別 |
| W2 `leo-aging-pressure` | pressure | 品質在掉，何時切 MEO？ | 切換時點 + 品質下降趨勢 |
| W3 `meo-continuity-hold` | continuity-hold | MEO 能撐多久？新 LEO 何時回？ | 持續時間 + LEO ETA + 覆蓋類別 |
| W4 `leo-reentry-candidate` | candidate | 新 LEO 回來，是否夠切回？ | 候選品質 + 切回後預期服務 |
| W5 `geo-continuity-guard` | guard | GEO 為何只是保底？序列何時結束？ | guard 角色 + sequence-end + boundary |

Viewer-facing body copy is pure Chinese. Small English technical tags
may remain only where they identify stable model concepts, for example
`MEO continuity-hold`, `V4.6D`, `TLE`, or `R2`.

### 4.2 Decision tab keeps a clean 3-section structure

Decision tab content has exactly three sections:
- 現在: 1-2 sentence Chinese state explanation.
- 為什麼: 1-2 sentence reason / pressure source.
- 下一步: 1 sentence next-state preview.

- A fourth `Watch` row.
- Engineering-side wording such as "modeled primary" in user-facing

controller. "Recommended action" verbs would over-claim.

### 4.3.1 Disabled metric tile interaction and accessibility

Disabled tiles are product explanations, not broken controls.

Contract:
- Render disabled metric tiles as non-interactive `article`/`li` tiles
  with `aria-disabled="true"` and no `onClick`.
- Do not put disabled tiles in tab order.
- Cursor is default or `not-allowed`, never pointer.
- Reduced opacity alone is insufficient; include an unavailable label,
  muted state token, and hookpoint copy.
- Placeholder values must be realistic and explicit, for example
  `未連接`, `Phase 7.1 gate`, or `external validation`.
- If a future route link is added, it must be a separate focusable link
  inside or below the tile, not the disabled tile itself. The link label
  must include the reachability qualifier.
- Tooltip/detail affordances, if any, use a separate info button with
  accessible name and `aria-describedby`; disabled state cannot depend on
  hover-only explanation.

### 4.3.2 Metric tile typography contract

Metric tiles use dashboard-data typography:
- numeric values, countdowns, and ratio-like strings use
  `--m8a-v411-font-data` and tabular figures;
- units are smaller label text, not part of the dominant numeric value;
- each tile has stable min width and stable value slot width so countdown
  changes do not shift layout;
- unavailable tiles reserve the same value slot using an em dash or
  Chinese unavailable label;
- value hierarchy is value > unit > label > hookpoint;
- no tile resizes on hover, focus, countdown tick, Archive expand, or
  tab switch.

### 4.4 Boundary ownership matrix

Boundary ownership follows the transfer principle: one visual channel,
one owner per subtype.

| Boundary subtype | Owner surface | Default visibility | Interaction path | Smoke assertion |
|---|---|---|---|---|
| Scale: `13-actor demo`, not >=500 LEO | Inspector boundary header strip | Visible whenever Details is open | Persistent above tab bar; no tab switch | Strip contains exact chip; no Decision body duplicate |
| Endpoint precision: `operator-family precision` | Inspector boundary header strip | Visible whenever Details is open | Persistent above tab bar; no tab switch | Strip contains exact chip; no Decision body duplicate |
| Simulation: modeled, not measured truth | Footer disclosure chip | Ambient visible | Footer chip opens full truth disclosure | Footer owns simulation wording |
| Service-layer: not native RF | Footer disclosure chip | Ambient visible | Footer chip opens full truth disclosure | Footer owns service-layer wording |
| Validation readiness / lock-in status | Validation status surface | Inspector header badge when Details open; footer/status chip when Details closed | Low-prominence badge; not top strip | No top-strip status slot; text budget obeyed |

Decision tab no longer owns boundary copy. The top strip keeps existing
scope/replay/precision/boundary slots until implementation migrates
copy, but v2 must not add validation status into the top strip.

### 4.4.x Inspector boundary header strip contract

The Scale + Endpoint boundary is an inspector header strip that sits
above the tab bar and below the sheet header. It is persistent across
Decision, Metrics, and Evidence.

Contract:
- Max height: `32px`.
- Single line; no wrapping at desktop and landscape tablet.
- Contains exactly two chips:
  - `13-actor demo`
  - `operator-family precision`
- It does not include validation readiness.
- It does not include simulation/service-layer disclosure.
- It must be present regardless of active inspector tab.
- It has its own DOM slot, for example
  `data-m8a-v411-inspector-boundary-strip="true"`.

Current DOM fact: the existing inspector header has title + Close, then
the tab bar. Implementation must add this strip between those two
regions.

### 4.5 Validation status badge contract

Validation readiness is low-prominence status, not boundary body copy.

When Details is open:
- render a compact badge in the inspector sheet header, near but not
  replacing the title or Close button;
- text budget: <=28 characters Chinese or <=32 ASCII characters;
- example copy: `Validation: pending` or `驗證狀態：待補`;
- do not use the top strip.

When Details is closed:
- represent the same status through a footer/status chip;
- this is the closed-state presentation of the same owner surface, not a
  second owner.

### 4.6 Evidence tab becomes summary plus Archive

Default Evidence content shrinks to two summary lines:
- `TLE: CelesTrak NORAD GP · 13 actors · fetched 2026-04-26`
- `R2: 5 candidate endpoints (read-only catalog)`

Archive expand reveals:
- 13 TLE row table.
- 5 R2 candidate endpoint table.
- Per-actor source details.

Default Archive state:
- W1-W4 collapsed.
- W5 expanded because sequence ending is audit-relevant.

No orbit token migration is authorized. Required migration is limited to:
1. Add `--m8a-v411-state-*` aliases centrally.
2. Migrate active/focus/disabled/warning state styling to state tokens.
3. Keep orbit identity usages on `--m8a-v411-orbit-*`.
4. Remove raw hex from component CSS except inside the central token file.
5. Update screenshots/smokes with a disclosure that state tokens changed
   channel ownership but orbit colors did not change.

Acceptance checks:
- no CSS rule uses orbit tokens for disabled state;
- no CSS rule uses state tokens to label orbit family;
- no component-level raw hex appears for new V4.11 state styling;
- reduced-motion mode preserves state meaning without pulse.

## 7. Reviewer mode state machine

Reviewer mode changes replay, inspector, toast, controls, and ARIA. It
must be specified as runtime state, not as a visual-only toggle.

### 7.1 Fields

Implementation must model:
- `replayClockMode: "running" | "review-auto-paused" | "inspector-pinned" | "manual-paused" | "final-hold"`
- `pinnedWindowId`: V4.6D window id captured at pause start
- `pinnedReplayRatio`: replay ratio captured at pause start
- `previousPlaybackState`: prior mode for restore
- `pauseSource: "review-mode-auto" | "inspector-open" | "user-pause-button"`
- `toastSuppressed: boolean`, true while inspector-pinned
- `reviewModeOn: boolean`, persisted user toggle state

### 7.2 Transitions

| From | Event | To | Required side effects |
|---|---|---|---|
| `running` | inspector-open click | `inspector-pinned` | capture `pinnedWindowId`, `pinnedReplayRatio`, `previousPlaybackState`; set `pauseSource="inspector-open"`; set `toastSuppressed=true`; pause replay |
| `inspector-pinned` | close | `running` or prior paused mode | default resume from `pinnedReplayRatio`; alternate resume from current clock time only behind explicit user toggle |
| `running` | `reviewModeOn=true` + window-transition | `review-auto-paused` | pause 8-12s; set `pauseSource="review-mode-auto"`; announce auto-pause |
| `review-auto-paused` | auto timer elapsed | `running` | resume playback from transition boundary |
| `review-auto-paused` | user action (pause, scrub, open inspector) | `manual-paused` or `inspector-pinned` | cancel auto-resume |
| `running` / paused | user pause button | `manual-paused` | set `pauseSource="user-pause-button"` |
| any active mode | final window end | `final-hold` | hold final frame; show restart cue |

Default policy:
- Review mode is default-on for customer review/demo/capture contexts.
- User choice persists.
- Inspector close resumes from pinned ratio by default.
- "Resume from current clock" is an explicit advanced toggle, not the
  default.

### 7.3 Toast behavior

While `inspector-pinned`:
- transition toasts are suppressed and queued only as internal state;
- suppressed toasts are discarded on inspector close;
- the reviewer must not see stale toasts replayed after closing;
- after close, the reviewer sees only toasts for new real-time
  transitions.

Toast is not a critical-only channel. Window identity and state must also
be visible in rail, sequence position, inspector header state, and ARIA.

### 7.4 Controls and labels

Control availability:
- `running`: Pause, Restart, Speed, Review toggle available.
- `review-auto-paused`: Resume, Pause, Restart, Review toggle available;
  speed changes apply after auto-resume unless user manually resumes.
- `inspector-pinned`: Close, Resume, Restart available; speed controls
  visible but disabled or explicitly marked "applies after resume".
- `manual-paused`: Play, Restart, Speed, Review toggle available.
- `final-hold`: Restart primary; Play restarts from W1.

Visible mode labels:
- `Pinned to W2` or Chinese equivalent in inspector header.
- `Auto-pause: review mode`.
- `Paused by user`.
- `Final hold · restart`.

### 7.5 ARIA

Mode changes announce through a `role="status"` live region:
- `Reviewer pinned to W2`
- `Resumed`
- `Auto-pause: review mode`

Announcements must be polite, not focus-stealing. The inspector modal /
sheet owns focus only when opened on narrow or when implementation
chooses modal semantics.

## 8. Narrow and tablet interaction contract

### 8.1 Breakpoint behavior

390x844 and portrait tablet are first-class review surfaces.

| Viewport | Layout |
|---|---|
| 390x844 phone | top scope row becomes compact horizontal strip; handover rail becomes drawer; Details opens full-screen modal |
| 768x1024 portrait tablet | closer to narrow: drawer rail + full-screen or near-full-screen inspector modal |
| 1024x768 landscape tablet | constrained desktop variant; rail can remain visible if it does not collide with scene/inspector |
| 1440x900 desktop | full top strip + rail + inspector + footer composition |

### 8.2 Narrow modal contract

When Details opens on narrow / portrait tablet:
- use a full-screen modal or near-full-screen sheet with
  `min-height: 100dvh`;
- respect safe-area insets top and bottom;
- lock background scroll;
- trap focus inside the modal;
- Escape and Close both close the modal;
- return focus to the Details trigger;
- scene remains visible only as non-interactive background context, or is
  hidden behind a strong scrim if legibility requires;
- no content depends on hover;
- primary touch targets are at least 44px.

### 8.3 Drawer and footer behavior

- Handover rail drawer is opened by a visible icon/text control in the
  compact strip.
- Drawer focus returns to its trigger on close.
- Footer chips may wrap to two rows; W5 warning/status chip stays visible
  in first row.
- Sequence rail simplifies to active + next + total dot count on phone.
- Countdown remains visible with stable width and tabular figures.

### 8.4 Required captures

Implementation must capture:
- 390x844 phone.
- 768x1024 portrait tablet.
- 1024x768 landscape tablet.
- 1440x900 desktop.

Reduced-motion and dynamic text scaling must be checked before final
visual acceptance.
